# Infrastructure Deployment Recommendations

## Executive Summary

This infrastructure plan was generated by Inframate based on analyzing your application's requirements and codebase structure.

### Recommended AWS Services:

Amazon EC2, Amazon S3, AWS Elastic Beanstalk, Amazon DocumentDB (with MongoDB compatibility), Amazon Route 53, Amazon CloudWatch, AWS Certificate Manager, Application Load Balancer, Amazon VPC, AWS IAM, AWS Secrets Manager, Amazon ECR (for future containerization)

## Detailed Recommendations

*   **Containerization (Future Step):**
    *   **Crucial Next Step:** Strongly recommend Dockerizing the application. This will provide consistency across environments, simplify deployments, and enable more advanced orchestration options (like ECS or EKS) in the future.
    *   Create a `Dockerfile` for your Node.js application.
    *   Store the Docker image in Amazon Elastic Container Registry (ECR).
*   **Application Deployment Platform:**
    *   **AWS Elastic Beanstalk:** This is a good starting point as it abstracts away much of the underlying infrastructure management.
        *   Use the "Node.js" platform if deploying the application directly without Docker.
        *   **Better:** Use the "Docker" platform on Elastic Beanstalk once the application is containerized. This allows Elastic Beanstalk to pull the image from ECR.
    *   It handles provisioning EC2 instances, load balancing, auto-scaling, and application health monitoring.
*   **Database:**
    *   **Amazon DocumentDB (with MongoDB compatibility):** This is AWS's managed MongoDB-compatible database service.
        *   Provides scalability, high availability (Multi-AZ deployments), and automated backups.
        *   Place DocumentDB instances in private subnets for security.
*   **Networking (VPC):**
    *   **Custom VPC:** Create a Virtual Private Cloud (VPC) for network isolation.
    *   **Subnets:**
        *   **Public Subnets:** For resources that need direct internet access (e.g., Application Load Balancer, NAT Gateways).
        *   **Private Subnets:** For application servers (EC2 instances managed by Beanstalk) and the DocumentDB cluster. This enhances security by preventing direct internet access to these resources.
    *   **NAT Gateway:** Place NAT Gateways in public subnets to allow instances in private subnets to access the internet for updates, package downloads, or external API calls, without being directly exposed.
    *   **Internet Gateway:** Attach to the VPC to allow communication between resources in public subnets and the internet.
*   **Static Assets:**
    *   The `public` directory suggests static assets (images, CSS, client-side JS).
    *   If this is a Single Page Application (SPA) or has significant static content, consider serving it from **Amazon S3** with **Amazon CloudFront** as a CDN for improved performance, lower latency, and SSL termination. Elastic Beanstalk can also serve these, but S3/CloudFront is often more optimal for purely static assets. If it's a Server-Side Rendered (SSR) application (e.g., Next.js), these assets might be served by the Node.js application itself.
*   **Domain & SSL/TLS:**
    *   **Amazon Route 53:** For managing your application's domain name.
    *   **AWS Certificate Manager (ACM):** For provisioning and managing free public SSL/TLS certificates to secure traffic to your application. Integrate this with the Application Load Balancer managed by Elastic Beanstalk.
*   **Security:**
    *   **IAM Roles:** Use IAM roles for EC2 instances (managed by Beanstalk) to grant them necessary permissions to other AWS services (e.g., S3, Secrets Manager) without hardcoding credentials.
    *   **Security Groups:** Act as virtual firewalls.
        *   ALB Security Group: Allow HTTP (80) and HTTPS (443) from the internet.
        *   Application Security Group (for Beanstalk EC2 instances): Allow traffic from the ALB Security Group on the application port (e.g., 3000, 8080).
        *   DocumentDB Security Group: Allow traffic from the Application Security Group on port 27017.
    *   **AWS Secrets Manager:** Store sensitive information like database credentials, API keys, etc. The application can retrieve these at runtime using an IAM role.
*   **Logging and Monitoring:**
    *   **Amazon CloudWatch:**
        *   Collect logs from Elastic Beanstalk instances and application logs.
        *   Monitor metrics for EC2 instances, Load Balancer, DocumentDB.
        *   Set up CloudWatch Alarms for critical metrics (e.g., CPU utilization, error rates).
*   **CI/CD (Future Step):**
    *   Implement a CI/CD pipeline using **AWS CodePipeline**, **AWS CodeBuild** (for building and testing, and potentially Dockerizing), and **AWS CodeDeploy** (or leverage Elastic Beanstalk's deployment mechanisms). This is critical for automating deployments and improving reliability.
*   **Scalability & Availability:**
    *   Elastic Beanstalk's Auto Scaling groups will handle scaling EC2 instances based on demand.
    *   Deploy DocumentDB in a Multi-AZ configuration for high availability.
    *   Deploy Elastic Beanstalk environment across multiple Availability Zones.
*   **The `.sql` file:** Investigate its purpose. If it's for a secondary relational database, you might need to provision Amazon RDS. If it's for seeding DocumentDB, it might need conversion or a custom script. For now, the primary focus is MongoDB.

## Estimated Monthly Costs

This is a rough estimate for a small, highly available setup suitable for a startup or a new application. Prices vary by region (using `us-east-1` as an example) and usage.
*   **Elastic Beanstalk (EC2 Instances & ALB):**
    *   2 x `t3.small` EC2 instances (for HA): ~$0.0208/hr * 2 * 730 hrs/month = ~$30.37/month
    *   Application Load Balancer: ~$0.0225/hr * 730 hrs/month + LCU cost (assume low traffic, ~0.5 LCU) = ~$16.43 + ~$3.60 = ~$20.03/month
    *   *Subtotal (Compute & LB): ~$50.40/month*
*   **Amazon DocumentDB (with MongoDB compatibility):**
    *   2 x `db.t3.medium` instances (smallest recommended for HA cluster): ~$0.088/hr * 2 * 730 hrs/month = ~$128.48/month
    *   Storage: 50 GB SSD storage: ~$0.10/GB-month * 50 GB = ~$5.00/month
    *   I/O: (Variable, assume $5 for light usage) = ~$5.00/month
    *   Backup Storage: (Assume 50GB) ~$0.021/GB-month * 50GB = ~$1.05/month
    *   *Subtotal (Database): ~$139.53/month*
*   **Networking (VPC, NAT Gateway):**
    *   NAT Gateway (1 for HA, though can share across AZs for cost saving initially): ~$0.045/hr * 730 hrs/month = ~$32.85/month
    *   NAT Gateway Data Processing (assume 10GB): ~$0.045/GB * 10 GB = ~$0.45/month
    *   *Subtotal (Networking): ~$33.30/month*
*   **Route 53:**
    *   1 Hosted Zone: ~$0.50/month
    *   Queries: (Negligible for low traffic)
    *   *Subtotal (DNS): ~$0.50/month*
*   **CloudWatch:**
    *   Logs (basic tier often covers initial needs): ~Free to $5/month
    *   Metrics & Alarms (basic): ~Free to $5/month
    *   *Subtotal (Monitoring): ~$0-$10/month*
*   **S3 (if used for static assets, build artifacts):**
    *   Storage (assume 5GB): ~$0.023/GB-month * 5 GB = ~$0.12/month
    *   Requests/Data Transfer: (Negligible for low traffic)
    *   *Subtotal (S3): ~$0.12/month*
*   **AWS Certificate Manager (ACM):**
    *   Public certificates are free.
    *   *Subtotal (SSL): $0.00/month*
**Estimated Total Monthly Cost: ~$223.85 - $233.85**
**Cost Optimization Notes:**
*   This assumes a highly available setup from the start (2 EC2 instances, 2 DocumentDB instances, NAT Gateway).
*   For a development/testing environment, you could reduce to 1 EC2 instance, 1 DocumentDB instance, and potentially skip the NAT Gateway if instances are in public subnets (not recommended for production).
*   Consider Reserved Instances or Savings Plans for EC2 and DocumentDB once usage patterns are stable to save significantly (30-60%).
*   Right-size instances based on actual load. `t3.small` and `db.t3.medium` are starting points.

*Note: These cost estimates are approximate and may vary based on usage patterns, region, and AWS pricing changes.*

## Deployment Instructions

1. **Prerequisites**:
   - AWS CLI configured with appropriate credentials
   - Terraform installed (v1.0.0+)

2. **Configuration**:
   - Update variables in `terraform.tfvars` to match your requirements
   - Review the Terraform files to ensure they meet your needs

3. **Deployment**:
   ```bash
   terraform init
   terraform plan
   terraform apply
   ```

4. **Cleanup**:
   ```bash
   terraform destroy
   ```

## Notes

- The Terraform files have been generated in the `terraform/` directory
- The configuration is based on the application requirements specified in `inframate.md`
- You may need to customize the Terraform files for your specific needs
